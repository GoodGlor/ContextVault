# i18n — English ⇄ Ukrainian (Ukrainian default) — design

- **Date:** 2026-07-22
- **Status:** Approved, ready to implement
- **Feature:** C of a three-feature request (A: HEIC #101; B: model dropdown #102; C: this).

## Goal

Make the whole SPA bilingual — English and Ukrainian — with **Ukrainian as the default**,
plus a language switcher available everywhere. Translate **all** user-facing strings in
one pass (no mixed-language screens).

## Decisions (from brainstorming)

- **Library:** **react-i18next** (+ i18next) — the React standard; handles interpolation,
  pluralization, and persistence.
- **Coverage:** everything, all at once.
- **Default:** Ukrainian; the choice persists in `localStorage` (`contextvault.locale`),
  mirroring the existing session-persistence pattern.

## Design

### Setup
- **`src/i18n/index.ts`** — configures i18next with `initReactI18next`, resources from
  `locales/en.json` + `locales/uk.json`, `lng` read from `localStorage` (default `uk`),
  `fallbackLng: "en"`, and a `languageChanged` handler that persists the choice.
- **`src/main.tsx`** imports `./i18n` once (side-effect init) so both the pre-login auth
  pages and the authenticated shell share the instance. No provider wrapper needed
  (the `initReactI18next` global instance backs `useTranslation`).

### Translation
- Every user-facing literal (JSX text, buttons, headings, labels, placeholders,
  aria-labels, status labels, inline fallback error messages) becomes `t("ns.key")`.
- Keys are grouped by namespace: `common`, `layout`, `nav`, `login`, `changePassword`,
  `acceptInvite`, `query`, `queryTurn`, `sourceList`, `answerText`, and one per admin page
  (`repositories`, `adminSources`, `users`, `insights`). ~150 keys, each with EN + UK.
- Pluralized counts (insights gap/user counts) use i18next `_one/_other` (EN) and
  `_one/_few/_many/_other` (UK) suffixes with `{{count}}`.
- **Not translated:** dynamic data (repo/user/source names, filenames, dates, numbers),
  API error *detail* strings (`err.detail`), CSS classes, element ids, the `accept`
  attribute, route paths. Status **CSS classes** stay the raw enum (`status-failed`);
  only the visible label is translated.

### Language switcher
- **`src/components/LanguageSwitcher.tsx`** — a compact `<select>` (Українська / English)
  calling `i18n.changeLanguage`. Rendered in the app header (`Layout`) and on each
  pre-login auth card (`LoginPage`, `ChangePasswordPage`, `AcceptInvitePage`) so the
  language can be changed before signing in.

## Testing
- **Unit:** the existing suite asserts on English strings. `src/test/setup.ts` pins the
  test locale to English (`i18n.changeLanguage("en")` before each test), so those
  assertions keep passing against the English catalog. Status-label assertions updated to
  the new capitalized labels. A new `LanguageSwitcher.test.tsx` verifies switching to
  Ukrainian re-renders translated text.
- **e2e:** both specs `addInitScript` to set `contextvault.locale = "en"` before
  navigating, so the English-string assertions hold against the real app (which defaults
  to Ukrainian). A throwaway check confirmed the default-Ukrainian login renders.

## Out of scope
- Additional languages beyond EN/UK.
- Locale-aware number/date formatting (values are rendered as-is).
- Translating backend-generated API messages (they remain English; surfaced via
  `err.detail`).
